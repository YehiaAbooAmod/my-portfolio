DHCP & Superscope Configuration

Project Overview

This project demonstrates the installation and configuration of the DHCP Server role, creation of multiple scopes, and integration of a Superscope to handle address exhaustion across subnets. It also includes the setup of a User Class named “inter”, used to give specific clients access to the correct gateway for internet connectivity through DHCP Policies.

Steps Completed

  • Installed and configured the DHCP Server role.
  • Created a main scope (10.0.0.0) and defined excluded ranges for servers and the gateway.
  • Configured a second scope (192.168.1.0) and grouped both into a Superscope.
  • Created a User Class (inter) to assign the real gateway for internet-enabled clients.
  • Applied DHCP policies and verified the client configurations via CMD.

Screenshots & Explanations

Installing the DHCP Server Role on Windows Server. This enables automatic IP address assignment and centralized network management.

Creating the first DHCP scope 10.0.0.0 with range 10.0.0.1–10.0.0.100. Excluded 10.0.0.1–10.0.0.20 for servers and 10.0.0.100 for the default gateway.

On the client machine, used ipconfig /renew in CMD to request an IP address from the DHCP server.

The client successfully received an IP, Gateway, and DNS from the DHCP server — confirming that the scope works correctly.

Created a second scope 192.168.1.0 with the same logic: 1–20 reserved for servers and 100 as the gateway. Both scopes were then grouped into a Superscope to ensure continuity when one range runs out.

The Superscope running successfully, distributing IP addresses across both subnets when required.

Created a custom User Class named “inter”. This class is used to identify special clients that need access to the real gateway for internet connectivity.

Applied a DHCP Policy to detect clients with the “inter” class and assign them the correct gateway that provides external internet access.

The policy confirms that any device using the inter class receives the proper gateway automatically from DHCP.

On the client side, the ipconfig /all command shows the correct IP, Gateway, and DNS values — confirming that the User Class and Policy were successfully applied.
